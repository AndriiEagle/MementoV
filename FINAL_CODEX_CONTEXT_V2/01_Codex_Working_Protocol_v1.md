# Codex Working Protocol v1

Purpose: tell future Codex sessions exactly how to work on this project without wasting cycles, losing context, or "improving" the product into scope-chaos.

## 1. Default Behavior

Codex should assume:

- the concept is already frozen enough to build;
- the main job is execution quality, not more brainstorming;
- the fastest way to fail is to add ambition faster than trust, usability, and safety;
- the right bias is `smaller, sharper, truer`.

## 2. Decision Hierarchy

When documents conflict, use this order:

1. `00_Context_Index_for_Codex_v2.md`
2. `01_Codex_Working_Protocol_v1.md`
3. `02_Launch_Order_and_Kill_Criteria_v1.md`
4. `03_Compliance_and_Founder_Risk_Gates_v1.md`
5. `Final_Concept_Document_v3.md`
6. `Final_Codex_Swarm_Development_Bible_v1.md`
7. `Swarm_Task_Packets_v1.md`

If a conflict still remains:
- protect `ECHO first`;
- protect `CHROMA -> ECHO`;
- protect elderly simplicity;
- protect night-first atmosphere;
- protect privacy and safety;
- cut scope rather than inventing a third path.

## 3. What Codex Must Do First in a Fresh Session

Before writing code or proposing architecture changes:

1. Read the V2 index.
2. Read the working protocol.
3. Read launch order and kill criteria.
4. Read compliance and founder-risk gates.
5. Read the development bible and task packets if implementation is requested.
6. Inspect the actual repo before assuming anything exists.

## 4. What Codex Must Not Do

Do not:

- rebrand the product without explicit user instruction;
- collapse it into a generic AI SaaS;
- turn `CHROMA` into a plain article;
- turn `ECHO` into a cold transcript tool;
- insert auth friction before value;
- optimize for DAU theater, session length addiction, or vanity engagement;
- assume native mobile is the next logical step;
- fork core OSS repos just because they look important;
- put self-hosted `LiveKit` media plane on `Railway`;
- let a memory SDK become the canonical biography database;
- assume legal-financial founder constraints are already solved;
- use "Verein", grants, or welfare assumptions as if they were product facts.

## 5. Product Standard for Good Decisions

Choose the option that:

- gets first value to a real user faster;
- increases trust and dignity;
- keeps the UI calmer for elderly users;
- makes the family-buyer path clearer;
- preserves the emotional bridge from `CHROMA` into `ECHO`;
- remains modular for future swarm work;
- reduces legal/safety ambiguity.

Reject the option that:

- creates more demo-theater than real value;
- creates more architecture than product;
- adds fragile dependencies for cosmetic reasons;
- depends on unverified legal or growth assumptions.

## 6. Build Bias

The right build bias is:

- web-first;
- PWA-friendly;
- DOM-first;
- modular;
- typed;
- fork thin shells, not core engines;
- reuse maintained upstreams where the savings are real;
- easy for multiple agents to edit safely;
- emotionally strong but technically boring where possible.

The wrong build bias is:

- graphics-first;
- app-store-first;
- full-engine-first;
- fork-everything-first;
- social-distribution-first;
- AI-magic-first.

## 7. Swarm Rules

If swarm work is requested:

- use `Swarm_Task_Packets_v1.md`;
- keep write ownership disjoint;
- do not spawn overlapping file ownership;
- do not let multiple workers invent different truths;
- freeze upstream and fork decisions before implementation spreads;
- lock prompts, safety rules, and design tokens early;
- prefer completing one usable vertical path over making five half-finished systems.

## 8. Research Rules

If research is requested:

- use `Deep_Research_Prompts_Critical_Points_v1.md`;
- treat Swiss sources and primary sources as highest trust;
- clearly separate fact from inference;
- end with `keep / change / cut`;
- if the finding weakens the concept, say so directly.

## 9. Quality Bar for Responses

A good Codex response for this project should:

- reflect the frozen truths;
- show awareness of the current repo state;
- reduce risk, not just add ideas;
- say what is core vs later;
- explain tradeoffs in plain language;
- avoid fake certainty when a risk still needs verification.

## 10. Exact Prompts the User Can Reuse

### A. Best prompt to start a build session

```text
Read FINAL_CODEX_CONTEXT_V2 in order and treat it as the active context pack. Do not reinvent the product. Inspect the repo first, then continue from the frozen concept and the development bible. Build only the next smallest emotionally complete step.
```

### B. Best prompt to start a swarm implementation session

```text
Read FINAL_CODEX_CONTEXT_V2 in order. Then use Final_Codex_Swarm_Development_Bible_v1.md and Swarm_Task_Packets_v1.md as the execution baseline. Launch only the packets needed for the next milestone, keep write ownership disjoint, and do not expand scope.
```

### C. Best prompt for a brutal repo audit

```text
Read FINAL_CODEX_CONTEXT_V2 in order. Then audit the current repo against the frozen concept, the development bible, and the kill criteria. Tell me what is strong, what is fake-progress, what is missing, and what should be cut before more code is written.
```

### D. Best prompt for safe refinement

```text
Read FINAL_CODEX_CONTEXT_V2 in order. Improve the project only where it materially increases trust, clarity, safety, usability, or first-value delivery. Do not make it bigger just to make it look more advanced.
```

## 11. Final Rule

If Codex is unsure between:

- bigger vs clearer;
- more features vs more truth;
- more spectacle vs more trust;
- more architecture vs more usability;

it should choose:

- clearer;
- truer;
- more trustworthy;
- more usable.
